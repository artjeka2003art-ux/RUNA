# RUNA — PHASE A.1 HARDENING

## Что это за этап

Phase A уже сделала главный поворот:
- после onboarding появился reveal сфер
- появился Life Map
- сферы стали редактируемыми
- появился Sphere Detail
- появился sphere chat

Теперь нужен короткий hardening-этап.

Цель:
**сделать Phase A глубже и честнее, чтобы сферы ощущались не как UI-объекты, а как реальные части жизни пользователя.**

Это НЕ новый большой редизайн.
Это НЕ Phase B с 3D-графом.
Это короткий продуктовый hardening.

---

## Главные проблемы текущей Phase A

### 1. Sphere chat ещё не до конца аналитически “сферический”
Сейчас UI знает, из какой сферы идёт разговор.
Но analyst/update flow должен тоже явно понимать sphere context.

Нужно сделать так, чтобы sphere chat в аналитике был явно привязан к:
- sphere_id
- sphere_name
- sphere-specific context

А не выглядел как просто ещё одно сообщение пользователя.

### 2. Add sphere flow пока слишком технический
Сейчас новая сфера создаётся слишком механически.

Нужно приблизить flow к продуктовой логике:
- пользователь создаёт тему
- попадает внутрь новой сферы
- AI помогает оформить смысл сферы
- после первого обмена сфера ощущается живой, а не пустой

### 3. Sphere description пока слишком слабая
Сфера должна иметь не только name, но и смысл.

Нужно сделать так, чтобы у сферы был:
- description / interpretation
- короткий human-readable смысл
- возможность уточнять этот смысл через sphere chat

### 4. Sphere Detail пока ещё слишком “объектный”
Он уже работает, но должен сильнее ощущаться как:
**“вот отдельная часть моей жизни, с которой я могу работать”**

---

## Что делаем в этом этапе

## 1. Усиливаем sphere chat analytics

### Что нужно сделать
Когда пользователь пишет в sphere chat, downstream логика должна явно знать:
- sphere_id
- sphere_name
- что это сообщение пришло из конкретной сферы

### Важно
Не ограничивайся только UI-level context.
Нужно, чтобы analyst/update flow тоже знал сферу.

### Как можно сделать
На выбор:
- передавать analyst отдельный `sphere_id` / `sphere_name`
- или специальный formatted message с меткой сферы
- или новый analyst mode для sphere chat

Лучший вариант: **явный параметр sphere context**, а не только текстовая эвристика.

### Что должно получиться
После sphere chat граф должен обновляться более осмысленно именно в контексте этой сферы, а не как после общего сообщения.

---

## 2. Доводим add sphere flow до продуктового вида

### Сейчас
Пользователь создаёт сферу по имени, и этого мало.

### Должно стать
Flow:
1. пользователь нажимает "Добавить сферу"
2. задаёт рабочее название / тему
3. сразу попадает в Sphere Detail
4. видит, что Runa помогает оформить эту сферу
5. первый AI-ответ помогает понять:
   - что это за сфера
   - почему она важна
   - с чем она связана
6. после первого обмена сфера уже ощущается наполненной

### Для MVP
Не нужно строить огромный wizard.
Достаточно сделать это через:
- redirect в новую сферу
- стартовое AI сообщение / generated intro
- обновление description после первого контакта

---

## 3. Усиливаем sphere description / interpretation

### Что нужно
У сферы должно быть не просто название, а короткий смысл.

Нужно добиться, чтобы в Sphere Detail было видно:
- что это за часть жизни
- почему она важна
- что сейчас на неё влияет

### Как сделать
Минимально допустимый вариант:
- description генерируется при sphere creation / first sphere chat
- затем может обновляться после sphere chat
- если description пока нет — честный empty state, но с AI-помощью для заполнения

### Не надо
- не придумывай descriptions на фронте
- не делай декоративный lorem ipsum
- не делай overly long text

Нужен короткий, живой, понятный смысл.

---

## 4. Усиливаем Sphere Detail UX

### Цель
Sphere Detail должен ощущаться как отдельная часть жизни, а не просто detail page сущности.

### Нужно добавить / улучшить
- более сильный summary block наверху
- блок “что сейчас происходит в этой сфере”
- блок “что влияет на неё”
- блок “куда можно сдвинуться”
- более тёплый и focused copy

### Важно
Не перегружай экран.
Сделай 2–3 сильных блока, а не 10 серых карточек.

---

## 5. Не трогаем сейчас Phase B

Сейчас НЕ делаем:
- 3D graph
- deep graph exploration
- sphere-to-sphere navigation через graph UI
- сложную новую визуализацию графа
- большой новый редизайн всего приложения

# Как думать об этой фазе

На каждом шаге проверяй вопрос:

**После этого изменения пользователь сильнее чувствует, что каждая сфера — это реальная часть его жизни?**

Если нет — не делай.

---

# Что должно получиться в конце

После Phase A.1 пользователь должен мочь:

1. создать новую сферу и сразу почувствовать, что она “ожила”
2. зайти в любую сферу и увидеть не только title, но и смысл
3. поговорить с AI внутри сферы и понимать, что система реально работает в контексте этой сферы
4. увидеть более сильный Sphere Detail, а не просто объектный экран

Если это достигнуто — hardening успешен.
