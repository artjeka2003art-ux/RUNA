# RUNA — Frontend Redesign Plan for Claude Code

## Зачем этот файл
Этот документ — рабочая карта для фронтенд-пересборки Runa.
Цель не в "красивом UI", а в том, чтобы интерфейс наконец начал передавать суть продукта:

**Runa = не админка и не AI toy. Это система ясности для человека, который застрял после провала, кризиса или выгорания.**

Главный критерий каждого изменения:
**После открытия продукта пользователю должно стать легче, потому что он понял, что делать дальше.**

---

## Главная продуктовая проблема текущей версии
Сейчас backend и core-логика уже сильнее, чем интерфейс.
Проблема в том, что фронт выглядит как internal tool / dev dashboard:

- слишком много инженерного ощущения
- слишком мало человеческой пользы на первом экране
- пользователь видит структуру системы, а не свой путь
- prediction выглядит как технический модуль, а не как жизненные маршруты
- check-in показывает внутреннюю механику вместо ощущения прогресса

Нужно не "добавить красоты", а **сменить интерфейсную логику продукта**.

---

## Новый продуктовый каркас
Навигация должна быть такой:

- **Today** — главный экран, куда человек возвращается каждый день
- **Check-in** — ежедневный разговор и обновление состояния
- **Path** — сценарии будущего как маршруты, а не как техничный prediction-модуль

Нужно уйти от логики:
- "вот дашборд"
- "вот граф"
- "вот prediction"

И перейти к логике:
- "где я сейчас"
- "что меня тянет вниз"
- "что меня может вытащить"
- "какой мой следующий шаг"
- "что будет, если я ничего не поменяю"

---

## Очень важно: что НЕ делать в этом спринте
Не трогать:

- backend архитектуру
- Neo4j / Redis / Zep / prediction math
- analyst / scenario / companion agents
- auth
- deploy
- Pinecone
- push notifications
- глубокий рефактор backend API

Можно делать только **минимальные фронтенд-адаптации данных**, если это нужно для UX.

---

## Перед началом работы — cleanup
Сначала очистить проект от мусора, который приехал в архиве:

1. удалить `frontend/node_modules`
2. удалить `frontend/dist`
3. удалить `__MACOSX`
4. затем в `frontend/` заново выполнить установку зависимостей

Если build/dev не стартует, сначала починить среду, только потом идти в UI.

## PRIORITY 1 — `frontend/src/App.tsx`

### Что это за файл
Корневой layout приложения, навигация, переключение экранов.

### Проблема сейчас
Сейчас App создаёт ощущение обычной тёмной админки:
- табы названы слишком сухо
- есть developer feeling
- иконки выглядят случайными
- структура не продаёт продукт

### Что нужно сделать
1. Переименовать табы:
   - `Главная` → `Today`
   - `Чекин` → `Check-in`
   - `Prediction` → `Path`

2. Убрать странные символы-иконки (`⬡ ◉ ◈`)
   - если нужны визуальные маркеры, сделать их очень минимальными
   - лучше вообще без декоративного мусора, чем с плохими иконками

3. Сделать навигацию более product-like:
   - аккуратная левая панель
   - сверху бренд `Runa`
   - снизу или рядом короткая фраза, например: `Clarity for today`
   - активная вкладка должна выглядеть как выбранный продуктовый режим, а не как пункт меню CRM

4. Сохранить простую структуру:
   - onboarding screen отдельно
   - после онбординга: Today / Check-in / Path

5. Не усложнять App бизнес-логикой
   - App должен только управлять shell/layout/navigation

### Цель изменения
App должен сразу визуально сказать: это не dev tool, а продукт для человека.

## PRIORITY 2 — `frontend/src/Dashboard.tsx`

### Что это за файл
Главный экран продукта. Самый важный файл всего фронта.

### Проблема сейчас
Сейчас Dashboard показывает:
- score ring
- количество узлов/связей
- граф личности
- списки сфер
- prediction preview

Это выглядит как демонстрация того, **как система устроена**, а не того, **чем она помогает пользователю**.

### Новая роль экрана
`Dashboard.tsx` нужно фактически превратить в **Today screen**.

### Что должно быть на экране
#### 1. Hero block
Большой верхний блок:
- Life Score
- человеческая интерпретация score
- короткое описание состояния

Примеры смыслов:
- `Ты в фазе восстановления. Система видит, что у тебя ещё много внутреннего шума, но часть опор уже собирается.`
- `Сейчас не точка силы, но и не хаос. Главное — не пытаться починить всю жизнь за один день.`

Важно: текст может быть rule-based на фронте из диапазона score, не обязательно тянуть новый backend.

#### 2. Three key spheres
Показать 3 самые важные сферы:
- название
- score
- краткий статус
- визуальный приоритет

Не просто список всех сфер одинаково.
Нужно выделить именно ключевые.

#### 3. "Что сейчас тянет вниз"
Отдельная карточка с главным blocker / blocker cluster.
Если есть несколько блокеров — показать 1 главный и максимум ещё 1 вторичный.

#### 4. "Что даёт опору"
Отдельная карточка с целями / ценностями / supportive element.
Не технично, а по-человечески.

#### 5. "Следующий шаг на сегодня"
Самый важный блок после hero.
Если backend не даёт готовый next step, собрать эвристически:
- по leverage point
- по weakest sphere
- по scenario preview

Формат:
- 1 конкретный следующий шаг
- 1 причина, почему именно он

#### 6. Small future preview
Небольшой блок:
- `Если оставить всё как есть`
- `Если сделать один честный шаг`

Это не полноценный Path screen, а тизер будущего.

#### 7. CTA
Явная кнопка/ссылка: `Сделать check-in`

### Что нужно убрать
Полностью убрать с главного экрана:
- mini graph SVG
- метрики `узлов`, `связей`
- визуализацию графа как центр внимания
- всё, что пахнет internal instrumentation

### Цель изменения
Today screen должен стать экраном, из-за которого человек вообще захочет возвращаться.

## PRIORITY 3 — `frontend/src/Chat.tsx`

### Что это за файл
Онбординг и ежедневный check-in чат.

### Проблема сейчас
В режиме check-in чат всё ещё выглядит слишком утилитарно, а после ответа показывает технические хвосты:
- `Граф: N изм.`
- `Life Score: X`

Это рушит ощущение живого продукта.

### Что нужно сделать
1. Сохранить два режима:
   - onboarding
   - check-in

2. Для режима `check-in` сделать более тёплую шапку:
   - заголовок
   - короткий подзаголовок
   - ощущение, что это ежедневная точка возвращения, а не просто input box

3. После check-in НЕ показывать сырые debug-like данные.
Вместо этого сделать человекочитаемый summary block:
- `Что улучшилось`
- `Что просело`
- `Как изменился Life Score`

4. Если `graph_updates` приходят сырые, преобразовать их на фронте в простой summary.
Даже если summary будет эвристическим, это лучше, чем raw update count.

5. Поле ввода и чат сделать более спокойными и премиальными:
- меньше ощущение обычного мессенджера
- больше ощущение guided reflection tool

6. Онбординг оставить относительно простым, но визуально единым с новым стилем продукта.

### Что НЕ делать
- не засорять чат аналитическими деталями
- не показывать внутренние термины graph/prediction math

### Цель изменения
Человек должен чувствовать: со мной разговаривают осмысленно, а не просто логируют мои сообщения.

## PRIORITY 4 — `frontend/src/PredictionView.tsx`

### Что это за файл
Экран со сценариями будущего.

### Проблема сейчас
Сейчас это выглядит как технический prediction-модуль:
- заголовок `Prediction`
- подпись `graph_math / 12w projection`
- карточки с numbers-first подачей

Это убивает магию.

### Новая роль экрана
`PredictionView.tsx` нужно превратить в **Path screen**.

### Что должно быть на экране
1. Новый заголовок:
- `Path`
- подзаголовок вроде `Три маршрута, по которым сейчас может пойти твоя жизнь`

2. Три сценария должны ощущаться как маршруты:
- текущий курс / realistic
- рост / optimistic
- откат / pessimistic

3. Для каждого сценария показать:
- human title
- вероятность
- короткое narrative summary
- score delta
- главный риск
- первый шаг

Если backend не даёт риск/первый шаг явно — вывести это эвристически из narrative / leverage / warning.

4. Вынести leverage point и warning signal в отдельные смысловые карточки:
- `Главная точка влияния`
- `Ранний сигнал риска`

5. Убрать из UI любые внутренние формулировки:
- `graph_math`
- `projection`
- `12w`
- любые инженерные подписи

6. Сделать карточки сценариев визуально разного веса:
- realistic как основной текущий путь
- optimistic как потенциальный рост
- pessimistic как риск-сценарий

### Цель изменения
Пользователь должен не "посмотреть prediction", а **увидеть свой путь и последствия бездействия**.

## PRIORITY 5 — `frontend/src/App.css`

### Что это за файл
Главный слой визуальной системы.

### Проблема сейчас
CSS рабочий, но интерфейс выглядит плоско и местами случайно:
- карточки слишком похожи друг на друга
- нет сильной типографической иерархии
- есть ощущение сырого dark dashboard
- важные блоки не доминируют

### Что нужно сделать
1. Перестроить визуальную иерархию:
- главный экран = больше воздуха
- hero крупнее
- secondary info слабее
- action blocks визуально сильнее

2. Сделать единый тёмный продуктовый стиль:
- глубокий тёмный фон
- мягкие границы
- аккуратные светы/градиенты
- без дешёвого неона

3. Сделать 3 уровня карточек:
- hero / primary
- secondary
- quiet/supporting

4. Улучшить typography:
- крупнее ключевые числа
- крупнее заголовки
- меньше серого мелкого текста
- лучше line-height

5. Улучшить spacing:
- больше вертикального дыхания
- меньше ощущения тесной панели

6. Привести chat, dashboard и path screen к одному стилю

### Важно
Не делать "дизайн ради дизайна".
Каждый визуальный выбор должен усиливать ощущение ясности.

# UI-логика, которую нужно держать в голове при работе

## Today screen отвечает на 5 вопросов
1. Где я сейчас?
2. Что меня больше всего тянет вниз?
3. Что у меня ещё живо и может стать опорой?
4. Какой мой следующий шаг?
5. Что будет, если я продолжу так же?

## Check-in отвечает на 3 вопроса
1. Как я сегодня?
2. Что реально изменилось?
3. Что система теперь понимает лучше?

## Path screen отвечает на 3 вопроса
1. Куда я иду сейчас?
2. Куда могу прийти, если соберусь?
3. Что будет, если ничего не менять?

---

# Ожидаемый результат после этого спринта
После изменений пользователь должен открыть Runa и почувствовать:

- это продукт про меня, а не про технологию
- система уже поняла мою ситуацию
- у меня есть не просто оценка, а направление
- мне хочется вернуться завтра

Если после редизайна интерфейс всё ещё выглядит как dashboard для фаундера, значит работа не закончена.
